4 # This module implements a Finite State Machine (FSM). In addition to state
5 # this FSM also maintains a user defined "memory". So this FSM can be used as a
6 # Push-down Automata (PDA) since a PDA is a FSM + memory.
7 #
8 # The following describes how the FSM works, but you will probably also need to
9 # see the example function to understand how the FSM is used in practice.
10 #
11 # You define an FSM by building tables of transitions. For a given input symbol
12 # the process() method uses these tables to decide what action to call and what
13 # the next state will be. The FSM has a table of transitions that associate:
14 #
15 # (input_symbol, current_state) --> (action, next_state)
16 #
17 # Where "action" is a function you define. The symbols and states can be any
18 # objects. You use the add_transition() and add_transition_list() methods to add
19 # to the transition table. The FSM also has a table of transitions that
20 # associate:
21 #
22 # (current_state) --> (action, next_state)
23 #
24 # You use the add_transition_any() method to add to this transition table. The
25 # FSM also has one default transition that is not associated with any specific
26 # input_symbol or state. You use the set_default_transition() method to set the
27 # default transition.
28 #
29 # When an action function is called it is passed a reference to the FSM. The
30 # action function may then access attributes of the FSM such as input_symbol,
31 # current_state, or "memory". The "memory" attribute can be any object that you
32 # want to pass along to the action functions. It is not used by the FSM itself.
33 # For parsing you would typically pass a list to be used as a stack.
34 #
35 # The processing sequence is as follows. The process() method is given an
36 # input_symbol to process. The FSM will search the table of transitions that
37 # associate:
38 #
39 # (input_symbol, current_state) --> (action, next_state)
40 #
41 # If the pair (input_symbol, current_state) is found then process() will call the
42 # associated action function and then set the current state to the next_state.
43 #
44 # If the FSM cannot find a match for (input_symbol, current_state) it will then
45 # search the table of transitions that associate:
46 #
47 # (current_state) --> (action, next_state)
48 #
49 # If the current_state is found then the process() method will call the
50 # associated action function and then set the current state to the next_state.
51 # Notice that this table lacks an input_symbol. It lets you define transitions
52 # for a current_state and ANY input_symbol. Hence, it is called the "any" table.
53 # Remember, it is always checked after first searching the table for a specific
54 # (input_symbol, current_state).
55 #
56 # For the case where the FSM did not match either of the previous two cases the
57 # FSM will try to use the default transition. If the default transition is
58 # defined then the process() method will call the associated action function and
59 # then set the current state to the next_state. This lets you define a default
60 # transition as a catch-all case. You can think of it as an exception handler.
61 # There can be only one default transition.
62 #
63 # Finally, if none of the previous cases are defined for an input_symbol and
64 # current_state then the FSM will raise an exception. This may be desirable, but
65 # you can always prevent this just by defining a default transition.
